\### Task 1 – Explanation of Each Model (in your own words)



\#### 1. Waterfall Model (Traditional)



\*\*What it is\*\*  

The Waterfall model is a linear, sequential software development process where each phase (requirements → design → implementation → testing → deployment → maintenance) must be fully completed before the next phase begins. There is no going back to a previous phase once it is finished.



\*\*When to use it\*\*  

It is best suited for projects with well-understood, stable requirements that are unlikely to change, such as safety-critical systems or projects with fixed contracts and strict regulatory compliance.



\*\*Real-world example\*\*  

NASA used the Waterfall model for the software of the Space Shuttle because requirements were fixed, safety was critical, and extensive documentation was required before any coding began.



\#### 2. Agile Methodology (Iterative / People-focused)



\*\*What it is\*\*  

Agile is a mindset and set of principles (Agile Manifesto) that values individuals and interactions, working software, customer collaboration, and responding to change over rigid processes and documentation. Development is done in small, iterative cycles called sprints or iterations.



\*\*When to use it\*\*  

It is ideal for projects with evolving or unclear requirements, where customer feedback is frequent and the team needs to adapt quickly – for example, web applications, mobile apps, and startup products.



\*\*Real-world example\*\*  

Spotify uses Agile methods (with a unique “Squad” model) to continuously release new features and adapt to user feedback without long planning phases.



\#### 3. Scrum Framework (Agile Framework)



\*\*What it is\*\*  

Scrum is a lightweight Agile framework that organizes work into fixed-length sprints (usually 2–4 weeks). It defines three roles (Product Owner, Scrum Master, Development Team), four ceremonies (Sprint Planning, Daily Stand-up, Sprint Review, Sprint Retrospective), and three artifacts (Product Backlog, Sprint Backlog, Increment).



\*\*When to use it\*\*  

Scrum works well for complex product development where requirements are not fully known at the start and the team can work cross-functionally. It is popular for software products that need frequent releases and stakeholder feedback.



\*\*Real-world example\*\*  

Google uses Scrum for many internal projects, including the development of certain Google Ads features, allowing them to deliver incremental value every sprint.



\#### 4. Kanban (Visual Workflow / Lean)



\*\*What it is\*\*  

Kanban is a visual workflow management method that uses a board (columns like “To Do”, “In Progress”, “Done”) and cards representing work items. It limits Work In Progress (WIP) to avoid overload and focuses on continuous delivery without fixed-length iterations.



\*\*When to use it\*\*  

It is best for maintenance projects, support teams, or continuous delivery pipelines where work arrives unpredictably (e.g., bug fixes, operational tasks) and you want to optimize flow and reduce lead time.



\*\*Real-world example\*\*  

Microsoft’s Visual Studio Team uses Kanban to manage bug fixes and feature requests, allowing them to respond quickly to issues without waiting for sprint boundaries.



\#### 5. DevOps (Culture + Practice)



\*\*What it is\*\*  

DevOps is a culture and set of practices that unifies software development (Dev) and IT operations (Ops). It emphasizes automation (CI/CD pipelines), continuous monitoring, infrastructure as code, and collaboration to shorten the development lifecycle and deliver reliable software rapidly.



\*\*When to use it\*\*  

DevOps is essential for cloud-native applications, microservices, and any project requiring frequent, reliable deployments – typically in organizations that need to release multiple times per day.



\*\*Real-world example\*\*  

Netflix uses DevOps practices extensively. Their engineers deploy code thousands of times per day using automated pipelines, and they run “Chaos Engineering” (e.g., Chaos Monkey) to test system resilience.



\#### 6. Spiral Model (Risk-Driven)



\*\*What it is\*\*  

The Spiral model combines elements of prototyping and the Waterfall model. Development proceeds in spirals (iterations), and each spiral includes four steps: determine objectives, identify and resolve risks, develop and test, and plan the next iteration. It focuses heavily on risk analysis.



\*\*When to use it\*\*  

It is best suited for large, high-risk, expensive projects where requirements are complex and evolving – for example, military systems, space exploration, or large enterprise software where failure is not acceptable.



\*\*Real-world example\*\*  

Boeing used the Spiral model for the development of the Boeing 777’s onboard software, because the project was large, required extensive risk management, and involved many unknown technical challenges.
